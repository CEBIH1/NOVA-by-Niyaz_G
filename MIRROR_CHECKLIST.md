# Чек-лист Зеркала (не тупить!)

Перед каждым ответом по проекту VeSkills проверять:

## 1. Роли и права
- [ ] `employee` — базовый класс. Все права на просмотр и оценку — от компетенции, а не от должности.
- [ ] `manager` = `employee` + админ. функции (управление подчинёнными). Никаких привилегий «потому что начальник».
- [ ] Роль «Мастер» в системе отсутствует. Есть «Руководитель» (manager) и уровень «Эксперт» (4).
- [ ] `trainer` = Ответственный за направление. Может не быть экспертом. Подтверждает Допуски/Обучения, для Скиллов — арбитраж.

## 2. Навыки (skills)
- [ ] Одна таблица `skills` с полем `type` (skill / permit / training).
- [ ] `validity_period` опционален (NULL = бессрочно). Не заставлять указывать срок для всего подряд.
- [ ] `has_levels` = true для skill, false для permit/training.

## 3. Данные
- [ ] `expiry_date` — вычисляемое поле (`granted_date + validity_period`). НЕ хранить в БД.
- [ ] `NULL` избегать. Для неприменимых уровней — `level_id = 0` («Не назначен»).

## 4. Верификация
- [ ] MVP: бинарная (подтверждено/отклонено) + обязательный комментарий при отклонении.
- [ ] Чек-листы — только в v2.0. Не усложнять старт.
- [ ] Trainer может подтверждать Допуски и Обучения. Для Скиллов — только арбитраж.
- [ ] Первый Эксперт назначается админом с яркой меткой «требуется проверка».

## 5. Пруфы
- [ ] Загружать может: Сотрудник, его Руководитель, Trainer. (Дядя Толя не умеет пользоваться компом).
- [ ] Лайкать/дизлайкать пруф может любой сотрудник с таким же навыком (равного или более высокого уровня).
- [ ] Оценивается не сотрудник, а его пруф. Уровень доверия складывается из оценок сообщества.
- [ ] Пруфы обязательны для публичного подтверждения навыка.

## 6. Оценка и уровни
- [ ] Уровень навыка не может быть снят руководителем. Только автоматически по сроку или через арбитраж.
- [ ] Новичок (уровень 1) видит всё, но оценивать не может. Оценка доступна с уровня 2 и выше.
- [ ] Руководитель назначает навык, но уровень выставляет Эксперт.
- [ ] `endorsements` — лайки/дизлайки с возможностью отзыва (revoked). В интерфейсе отозванное не показывается.

## 7. Терминология
- [ ] Не использовать слово «Мастер» как роль в системе.
- [ ] Не называть «Trainer» «Мастером» или «Смотрителем».
- [ ] Не называть «Руководителя» (manager) «Мастером».
- [ ] «Парампара» — это цепочка поручительства Экспертов, а не административная верификация.

## 8. Общение и формат работы
- [ ] Не предлагать сигареты и другие вредные вещи.
- [ ] Не использовать пассивно-агрессивные формулировки («Теперь понятно?»).
- [ ] Не называть Нияза чайником или новичком.
- [ ] Не использовать сленг вроде «Нияз Edition» для инструкций.
- [ ] Нияз не может отправлять файлы. Не просить прислать скриншоты, файлы, схемы. Работать только через текстовые описания и код.
- [ ] **Если Нияз прислал текущую версию файла — править ТОЛЬКО то, что он просил. Остальной код не трогать. Возвращать ровно тот же файл с точечными изменениями. Не генерировать файл заново.**

## 9. Технический стек
- [ ] Бэкенд: Node.js + Express + pg (чистый SQL, параметризованные запросы).
- [ ] Фронтенд: Vue 3 (Composition API) + PrimeVue + FontAwesome Solid.
- [ ] БД: PostgreSQL. Миграции в папке `migrations/`.
- [ ] Деплой: Docker, on-premise, без облаков.

## 10. Ключевые файлы проекта
- [ ] `README.md` — описание проекта.
- [ ] `VISION.md` — концепция, границы, роли, принципы.
- [ ] `ROADMAP.md` — план версий и фич.
- [ ] `MIRROR_CHECKLIST.md` — этот файл.
- [ ] `CONTEXT.txt` — строка для перезапуска диалога.
- [ ] `roles.docx` — ролевая модель.
- [ ] `migrations/` — SQL-файлы миграций.
